# Serendipity - 开发路线图

> **项目名称**：错过了么 / Serendipity  
> **创建日期**：2026-02-09  
> **文档版本**：v1.0

---

## 📋 技术栈确认

### 前端
- **框架**：Flutter 3.x
- **语言**：Dart 3.x
- **状态管理**：Riverpod 2.x
- **路由**：go_router
- **本地存储**：Hive
- **地图服务**：高德地图 API (`amap_flutter_map`)

### 后端
- **方案**：Firebase（纯云服务）
- **认证**：Firebase Authentication
- **数据库**：Cloud Firestore
- **云函数**：Cloud Functions
- **推送**：Firebase Cloud Messaging (FCM)
- **存储**：暂不使用 Firebase Storage（后期如需头像上传功能再添加）

### 平台
- ✅ Android（最低支持 API 21 / Android 5.0+）
- ✅ iOS（最低支持 iOS 12.0+）

#### 4.2 匹配算法（5-7天）
- [ ] 数据模型
  - `Match`
  - `MatchCandidate`
  - `CandidateRecord`
  - `MatchStatus` 枚举
  - `MatchConfidence` 枚举
  - `VerificationChoice` 枚举
  - `UserCreditScore`
  - `CreditChange`
  - `KeepInMemoryList`
- [ ] 五重防御机制
  - 第一重：对方也使用 Serendipity
  - 第二重：对方也记录了那次错过
  - 第三重：双方都将记录加入了故事线（技术要求）
  - 第四重：GPS 距离 < 200米 + 时间差 < 15分钟 + 地点名称模糊匹配
  - 第五重：标签的双向验证（防骗核心）
- [ ] 匹配逻辑（Cloud Functions）
  - 用户创建记录时触发
  - 只匹配故事线中的记录
  - 按用户分组，选择评分最高的记录
  - 收集所有匹配的记录ID（按评分排序）
  - 检查是否已有活跃的Match（避免重复通知）
  - 6小时冷静期
  - 推送两种通知
- [ ] 两种通知类型
  - 类型1：你的记录匹配到了候选人
  - 类型2：有人选择了你的记录
- [ ] 双向验证流程
  - 用户A收到通知，查看候选记录列表
  - 用户A选择"哪个是我"（展示所有候选记录）
  - 三个选择：想要联系 / 留在记忆里 / 认错人了
  - 通知用户B（被选中的记录所有者）
  - 用户B确认身份
  - 双方都确认 + 都选择"想要联系" → 验证成功
- [ ] 匹配置信度
  - 高置信度：GPS < 50米 && 时间差 < 5分钟
  - 中置信度：GPS < 200米 && 时间差 < 15分钟
  - 低置信度：仅地点名称匹配（相似度 > 70%）
- [ ] 验证失败处理
  - 认错人了：温柔提示，未来可再次匹配
  - 留在记忆里：永久不再匹配，加入KeepInMemoryList
  - 双方选择的记录不一致：温柔提示
- [ ] 反作弊机制
  - GPS 真实性验证（坐标合理性、突然跳跃、模拟器检测）
  - 异常行为模式检测（不同地点高频、长期高频）
  - 速度检测（> 120km/h）
  - 信用评分系统（GPS异常 -30分，行为异常 -10分）
  - 信用分 < 0 禁止匹配功能

## 🚨 风险与应对

### 风险1：Firebase 在中国访问慢
**应对**：
- 使用 Firebase 国内镜像
- 添加本地缓存（Hive）
- 优化数据同步策略

### 风险2：高德地图 API 配额不够
**应对**：
- 监控 API 调用量
- 添加本地缓存
- 必要时升级付费套餐

### 风险3：匹配算法性能问题
**应对**：
- 使用 Cloud Functions 后台处理
- 添加索引优化查询（按 GPS、时间、storyLineId）
- 限制匹配频率
- 只匹配故事线中的记录（减少查询量）
- 候选人数量限制（最多10个）

### 风险4：支付集成复杂
**应对**：
- 使用成熟的支付插件
- 参考官方文档
- 先做 iOS（IAP 相对简单）

### 风险5：审核被拒
**应对**：
- 仔细阅读应用商店规则
- 准备详细的隐私政策
- 说明社交功能的安全机制
