# Serendipity - 项目规格文档

> **错过了么** - 记录通勤中那些擦肩而过的瞬间

## 📋 项目概述

### 项目信息
- **项目名称（中文）**：错过了么
- **项目名称（英文）**：Serendipity
- **项目类型**：移动应用（Android + iOS）
- **技术栈**：Flutter + Dart
- **目标用户**：18-35岁，经常通勤的都市青年
- **核心定位**：情感记录类应用，专注于记录通勤期间错过的人

### 项目愿景
在快节奏的都市生活中，我们每天都会与无数陌生人擦肩而过。有些人只是一瞥，却在心中留下涟漪。Serendipity 让用户记录这些转瞬即逝的瞬间，保存那些未曾开口的遗憾，也许某天，命运会给你第二次机会。

## 🎯 核心功能

### 1. 记录系统

#### 1.1 快速记录
用户可以快速记录一次"错过"，包含以下元素：

**必填项**：
- ⏰ **时间**：自动获取当前时间（可手动调整）
- 📍 **地点**：自动获取GPS定位（可手动输入）
- 💫 **状态**：选择当前状态（允许从任意状态开始）
  - 🌫️ 错过（第一次看到，但没说话）
  - 🌟 再遇（又看到了，但还是没说话）
  - 💫 邂逅（终于说话了）
  - 💝 重逢（说过话后，又见面了）
  - 🥀 别离（主动结束了）
  - 🍂 失联（被动消失，再也没见过）

**设计说明**：
- **允许从任意状态开始记录**：用户可能在开始使用 app 之前就已经经历了多个阶段
  - 例如：用户今天和某人分手了，想记录"别离"状态，之前的"错过"、"再遇"、"邂逅"、"重逢"都没记录
  - 例如：用户今天又在地铁上看到昨天那个人，选择"再遇"状态，昨天的"错过"没记录
- **每次状态变化 = 创建新记录**：如果又见面了，创建一个新记录，选择新状态
- **状态可以修改**：防止用户手滑点错状态，可以在记录详情中修改

#### 1.2 视觉设计原则
**❌ 不使用照片功能**，原因：
- 避免给用户心理压力（匆忙中拍照）
- 避免偷拍的道德和法律风险
- 保持"错过"的朦胧美感
- 保护隐私

**✅ 替代方案**：
- 使用渐变色卡片代表不同情感
- 用抽象图形/纹理增加美感
- 根据标签和描述生成意境插画（可选）

**显示优先级规则**：

```dart
/// 获取地点的显示文本
String getDisplayLocation(Location location) {
  // 优先级 1：用户输入的地点名称（最有温度）
  if (location.placeName != null && location.placeName!.isNotEmpty) {
    // 如果有场所类型，显示图标
    if (location.placeType != null) {
      return '${location.placeType!.icon} ${location.placeName}';
    }
    return location.placeName!;
  }
  
  // 优先级 2：GPS 获取的地址（标准但冷冰冰）
  if (location.address != null && location.address!.isNotEmpty) {
    // 如果地址太长，截断显示
    if (location.address!.length > 20) {
      return '${location.address!.substring(0, 20)}...';
    }
    return location.address!;
  }
  
  // 优先级 3：场所类型（至少有个分类）
  if (location.placeType != null) {
    return '${location.placeType!.icon} ${location.placeType!.label}';
  }
  
  // 优先级 4：实在没有，显示默认文本
  return '未知地点';
}
```

**设计说明**：

### 2. 状态流转系统

#### 2.1 六种状态

| 状态 | 英文 | 图标 | 含义 | 触发方式 |
|------|------|------|------|---------|
| **错过** | Missed | 🌫️ | 初次看到但未交流 | 创建记录 |
| **再遇** | Re-encounter | 🌟 | 再次看到同一个人 | 手动标记 |
| **邂逅** | Met | 💫 | 终于有了交流 | 手动标记 |
| **重逢** | Reunion | 💝 | 交流后再次见面 | 手动标记 |
| **别离** | Farewell | 🥀 | 主动结束关系（需先邂逅） | 手动标记 |
| **失联** | Lost | 🍂 | 被动消失，再也没见过 | 手动标记 |

#### 2.2 状态流转规则

```
[错过] → [再遇] → [再遇] → ... → [邂逅] ⇄ [重逢] ⇄ [别离]
  ↓        ↓        ↓               ↓         ↓         ↓
[失联]   [失联]   [失联]          [失联]    [失联]    [失联]
```

**规则说明**：
- **错过 → 再遇 → 再遇 → ...**：可以多次"再遇"（一直看到但没说话）
  - 例如：每天地铁上看到同一个人，看了10次、20次
  - 每次"再遇"都可以记录新的细节和心情
  - 直到某天终于"邂逅"（说话了）
- **邂逅 ⇄ 重逢 ⇄ 别离**：可以循环（交流后的人生）
  - 邂逅 → 重逢 ✅（又见面了）
  - 邂逅 → 别离 ✅（交流后分开）
  - 重逢 → 别离 ✅（见面后分开）
  - 别离 → 重逢 ✅（分手后复合）
  - 别离 → 邂逅 ✅（分开后重新开始）
- **失联**：唯一的终点，任何状态都可以标记为"失联"
- 每次状态变更都会记录时间和备注

**别离 vs 失联**：
- **别离**：主动的结束（分手、好聚好散、发现不合适）
- **失联**：被动的消失（再也没见过、联系不上了）

**为什么允许循环？**
- 因为现实人生就是循环的
- 在一起 → 分手 → 复合 → 再分手...
- 这才是真实的人生

### 6. 社交功能

#### 6.1 设计理念

**核心概念**：这是一个"记录错过"的 app，但相信奇迹。

**产品哲学**：
- 主要功能：情感记录（99.99%）
- 匹配功能：奇迹匹配（0.01%）

**相信奇迹，但不强求**：
- 用户知道有匹配功能（产品营销卖点）
- 但后台静默运行，用户不知道"正在匹配中"（避免焦虑）
- 如果根本没匹配上，用户不会收到任何通知（避免失望）
- 只在匹配成功时推送通知："也许，TA 就在这里"
- 进入验证阶段后，如果验证失败，会温柔地告知用户

**树洞功能**：
- 用户可以选择将记录匿名发布到社区
- 其他用户可以浏览，但**无法互动**
- 没有评论、点赞功能
- 只能默默浏览

#### 6.2 匹配功能

**设计原则**：
- 用户知道有匹配功能（产品营销卖点）
- 但后台静默运行，用户不知道"正在匹配中"
- 如果根本没匹配上，用户不会收到任何通知
- 只在匹配成功时推送通知，进入验证阶段
- 验证阶段如果失败，会温柔地告知用户

**匹配条件（五重防御）**：

```
第一重：对方也使用 Serendipity
第二重：对方也记录了那次错过
第三重：双方都将记录加入了故事线（技术要求）
第四重：GPS 距离 < 200米 + 时间差 < 15分钟 + 地点名称模糊匹配
第五重：标签的双向验证（防骗核心）

→ 成功率：0.01%
→ 遇到骗子的可能性：极低
```

**关于第三重防御的说明**：

**为什么只有故事线中的记录参与匹配？**

**技术原因**：
- 程序无法自动判断哪些记录是同一个人
- 需要用户通过故事线告诉程序"这些记录是同一个人"
- 这样才能准确匹配，避免混淆

**举例说明**：
```
场景：用户A在地铁上看到3个人

用户A创建了3条记录：
- 记录1：长发女生
- 记录2：短发女生
- 记录3：戴眼镜男生

问题：程序不知道这3条记录是3个不同的人

解决：用户通过故事线告诉程序
- 记录1 → 故事线"地铁上的她"
- 记录2 → 故事线"咖啡馆的她"
- 记录3 → 故事线"图书馆的他"

结果：程序知道这是3个不同的人，可以准确匹配
```

**类比**：
- 就像QQ添加好友需要输入QQ号
- 不是给你展示几亿用户让你海选
- 创建故事线是帮助程序识别"哪些记录是同一个人"

**关于第四重防御的说明**：

**现实问题**：
- 用户通常在到达公司/家里后才记录（通勤路上不方便看手机）
- 导致GPS显示的是家/公司地址，而非实际错过的地点
- 这是物理上的技术隔阂，无法完全解决

**解决方案（三管齐下）**：

1. **放宽GPS距离阈值**：从 50米 → 200米
   - 原因：用户可能在附近才记录
   - 例如：在地铁站错过，走出站后才记录
   - 200米可以覆盖大部分"附近记录"的场景

2. **放宽时间差阈值**：从 5分钟 → 15分钟
   - 原因：用户可能需要时间找到合适的地方记录
   - 例如：18:00错过，18:10到公司，18:15才打开app记录
   - 15分钟可以覆盖大部分"延迟记录"的场景

3. **地点名称模糊匹配**：
   - 如果GPS距离 > 200米，但地点名称相似度 > 70%，也算匹配
   - 例如：
     - 用户A：GPS在家，但手动输入"地铁10号线国贸站"
     - 用户B：GPS在公司，但手动输入"地铁10号线 国贸"
     - 模糊匹配：相似度 85% → 匹配成功
   - 使用 Levenshtein 距离算法计算相似度
   - 忽略空格、标点符号、大小写

**匹配逻辑（优先级）**：

```
优先级1：GPS精确匹配（最可靠）
- GPS距离 < 50米
- 时间差 < 5分钟
→ 直接匹配成功

优先级2：GPS宽松匹配（次可靠）
- GPS距离 < 200米
- 时间差 < 15分钟
→ 匹配成功

优先级3：地点名称模糊匹配（补充方案）
- GPS距离 > 200米（GPS不准确）
- 但地点名称相似度 > 70%
- 且时间差 < 15分钟
→ 匹配成功（但标记为"低置信度"）

优先级4：完全不匹配
- GPS距离 > 200米
- 地点名称相似度 < 70%
→ 匹配失败
```

**为什么这样设计？**

1. **接受现实**：
   - 用户不可能在通勤路上立即记录（物理限制）
   - 必须放宽匹配条件，否则成功率接近0%

2. **保持安全**：
   - 即使放宽到200米、15分钟，仍然很难作弊
   - 骗子无法预测受害者的精确时间和地点
   - 第四重防御（标签验证）仍然是核心

3. **地点名称作为补充**：
   - 如果用户手动输入了地点名称，说明记忆清晰
   - 两个人都记得"地铁10号线国贸站"，即使GPS不准，也很可能是真的
   - 模糊匹配允许轻微差异（如"国贸站" vs "国贸"）

4. **低置信度标记**：
   - 如果只通过地点名称匹配（GPS不准），标记为"低置信度"
   - 在验证页面提示用户："GPS定位不准确，请仔细核对信息"
   - 降低误判风险

**匹配算法伪代码**：

```dart
/// 为一条记录查找所有匹配的候选人
List<MatchCandidate> findMatchCandidates(Record myRecord) {
  List<MatchCandidate> candidates = [];
  
  // 0. 检查：这条记录是否属于故事线？
  if (myRecord.storyLineId == null) {
    return [];  // 不属于故事线，不参与匹配
  }
  
  // 1. 查询数据库：时间范围内的所有记录（且属于故事线）
  List<Record> potentialRecords = queryRecordsInTimeRange(
    myRecord.timestamp - 15分钟,
    myRecord.timestamp + 15分钟,
    onlyStoryLineRecords: true  // 只查询属于故事线的记录
  );
  
  // 2. 按用户分组（同一个用户的多条记录归为一组）
  Map<String, List<Record>> recordsByUser = {};
  for (Record record in potentialRecords) {
    // 跳过自己的记录
    if (record.userId == myRecord.userId) continue;
    
    if (!recordsByUser.containsKey(record.userId)) {
      recordsByUser[record.userId] = [];
    }
    recordsByUser[record.userId].add(record);
  }
  
  // 3. 对每个用户，选择评分最高的记录作为代表
  for (String userId in recordsByUser.keys) {
    // 检查：是否已经有活跃的Match？
    if (hasActiveMatch(myRecord.userId, userId)) {
      continue;  // 已经有活跃Match，跳过（避免重复通知）
    }
    
    List<Record> userRecords = recordsByUser[userId];
    
    MatchCandidate? bestCandidate = null;
    double bestScore = 0;
    
    // 遍历该用户的所有记录，找到评分最高的
    for (Record otherRecord in userRecords) {
      // 计算时间差
      int timeDiff = abs(myRecord.timestamp - otherRecord.timestamp);
      if (timeDiff > 15分钟) continue;
      
      // 计算GPS距离
      double distance = calculateDistance(
        myRecord.latitude, myRecord.longitude,
        otherRecord.latitude, otherRecord.longitude
      );
      
      // 计算地点名称相似度
      double placeNameSimilarity = 0.0;
      if (myRecord.placeName != null && otherRecord.placeName != null) {
        placeNameSimilarity = calculateSimilarity(
          myRecord.placeName,
          otherRecord.placeName
        );
      }
      
      // 计算标签相似度（用于排序，不用于过滤）
      double tagSimilarity = calculateTagSimilarity(
        myRecord.tags,
        otherRecord.tags
      );
      
      // 判断是否匹配
      bool isMatch = false;
      String matchType = '';
      
      // 优先级1：GPS精确匹配
      if (distance < 50米 && timeDiff < 5分钟) {
        isMatch = true;
        matchType = 'high_confidence';
      }
      // 优先级2：GPS宽松匹配
      else if (distance < 200米 && timeDiff < 15分钟) {
        isMatch = true;
        matchType = 'medium_confidence';
      }
      // 优先级3：地点名称模糊匹配
      else if (placeNameSimilarity > 0.7 && timeDiff < 15分钟) {
        isMatch = true;
        matchType = 'low_confidence';
      }
      
      // 如果匹配，计算评分
      if (isMatch) {
        double score = calculateScore(distance, timeDiff, tagSimilarity);
        
        // 如果这条记录的评分更高，更新最佳候选
        if (score > bestScore) {
          bestScore = score;
          bestCandidate = MatchCandidate(
            userId: userId,
            record: otherRecord,
            matchType: matchType,
            distance: distance,
            timeDiff: timeDiff,
            placeNameSimilarity: placeNameSimilarity,
            tagSimilarity: tagSimilarity,
            score: score,
            alternativeRecordCount: 0  // 稍后更新
          );
        }
      }
    }
    
    // 如果找到了最佳候选，加入列表
    if (bestCandidate != null) {
      // 收集所有匹配的记录ID（按评分排序）
      List<RecordWithScore> allMatchedRecords = [];
      
      for (Record record in userRecords) {
        // 重新计算每条记录的匹配情况
        int timeDiff = abs(myRecord.timestamp - record.timestamp);
        if (timeDiff > 15分钟) continue;
        
        double distance = calculateDistance(
          myRecord.latitude, myRecord.longitude,
          record.latitude, record.longitude
        );
        
        double placeNameSimilarity = 0.0;
        if (myRecord.placeName != null && record.placeName != null) {
          placeNameSimilarity = calculateSimilarity(
            myRecord.placeName,
            record.placeName
          );
        }
        
        double tagSimilarity = calculateTagSimilarity(
          myRecord.tags,
          record.tags
        );
        
        // 判断是否匹配
        bool isMatch = false;
        if (distance < 50米 && timeDiff < 5分钟) {
          isMatch = true;
        } else if (distance < 200米 && timeDiff < 15分钟) {
          isMatch = true;
        } else if (placeNameSimilarity > 0.7 && timeDiff < 15分钟) {
          isMatch = true;
        }
        
        if (isMatch) {
          double score = calculateScore(distance, timeDiff, tagSimilarity);
          allMatchedRecords.add(RecordWithScore(record, score));
        }
      }
      
      // 按评分排序
      allMatchedRecords.sort((a, b) => b.score.compareTo(a.score));
      
      // 提取记录ID列表
      List<String> candidateRecordIds = allMatchedRecords
        .map((r) => r.record.id)
        .toList();
      
      bestCandidate.candidateRecordIds = candidateRecordIds;
      candidates.add(bestCandidate);
    }
  }
  
  // 4. 按综合评分排序（分数高的在前）
  candidates.sort((a, b) => b.score.compareTo(a.score));
  
  // 5. 限制候选人数量（最多10个）
  if (candidates.length > 10) {
    candidates = candidates.sublist(0, 10);
  }
  
  return candidates;
}

/// 计算综合评分
double calculateScore(double distance, int timeDiff, double tagSimilarity) {
  // 距离分数：距离越近分数越高（0-100分）
  double distanceScore = max(0, 100 - distance / 2);
  
  // 时间分数：时间差越小分数越高（0-100分）
  double timeScore = max(0, 100 - timeDiff / 0.15);
  
  // 标签分数：相似度越高分数越高（0-100分）
  double tagScore = tagSimilarity * 100;
  
  // 加权平均：GPS距离最重要，时间次之，标签最后
  return distanceScore * 0.5 + timeScore * 0.3 + tagScore * 0.2;
}
```

**关键设计说明**：

1. **只有故事线中的记录参与匹配**：
   - 第一步检查：`myRecord.storyLineId == null` → 直接返回空列表
   - 查询时只查询属于故事线的记录：`onlyStoryLineRecords: true`
   - **原因**：程序需要知道哪些记录是同一个人，才能准确匹配
   - **技术限制**：程序无法自动判断，需要用户通过故事线告诉程序

2. **按用户分组**：
   - 同一个用户的多条记录归为一组
   - 避免同一个人出现多次

3. **收集所有匹配的记录**：
   - 对每个用户，收集所有匹配的记录（不只是最佳的）
   - 按评分排序（评分高的在前）
   - 保存到 `candidateRecordIds` 列表

4. **去重检查**：
   - 检查是否已经有活跃的Match（避免重复通知）
   - 活跃的Match包括：pending、notified、verifying、verified

5. **验证页面展示**：
   - 展示所有候选记录（按评分排序）
   - 让用户选择"哪个是我"
   - 用户可以选择其中一个，或选择"都不是我"

6. **为什么展示所有记录？**：
   - 评分最高不代表就是那个人
   - 可能GPS距离更近，但不是同一个人
   - 让用户自己判断，避免误判

### 活跃的Match检查

```dart
/// 检查两个用户是否已经有活跃的Match
bool hasActiveMatch(String userAId, String userBId) {
  Match? match = queryMatch(userAId, userBId);
  
  if (match == null) return false;
  
  // 以下状态算"活跃"
  return match.status == MatchStatus.pending ||      // 等待冷静期
         match.status == MatchStatus.notified ||     // 已通知
         match.status == MatchStatus.verifying ||    // 验证中
         match.status == MatchStatus.verified;       // 验证成功（正在聊天）
}
```

**说明**：
- 如果已经有活跃的Match，不再创建新的Match
- 避免同一对用户收到多次通知
- 只有在以下情况才会再次匹配：
  - `rejected` + `!isPermanentlyKeptInMemory`（认错人了）
  - `expired`（对话已过期）

- 高置信度匹配（GPS < 50米）：通知标题显示"🌟 奇迹发生了"，提示"GPS定位精确匹配"
- 低置信度匹配（仅地点名称匹配）：通知标题显示"🌟 可能的匹配"，提示"⚠️ GPS定位不准确，请仔细核对信息"

**多候选人匹配说明**：

**场景**：用户 A 的一条记录匹配到了多个候选人（用户 B、C、D）

**处理方式**：
1. **全部匹配**：创建多个 Match 记录（A-B、A-C、A-D）
2. **依次通知**：6小时后，同时推送通知给所有匹配对
3. **独立验证**：每个匹配对独立验证，互不影响
4. **可能的结果**：
   - 所有人都验证失败 → 没找到
   - 只有一个验证成功 → 找到了
   - 多个验证成功 → 用户自己选择和谁聊天

**为什么这样设计？**
- ✅ 不错过任何可能性（也许真的是其中一个）
- ✅ 让用户自己判断（通过标签验证）
- ✅ 避免算法误判（不由系统决定"最像的那个"）
- ✅ 真实场景：地铁上可能有多个人符合描述

**用户体验**：
- 用户会收到多条"奇迹发生了"通知
- 每条通知对应一个候选人
- 用户依次查看、验证
- 找到真正的那个人

**极端情况处理**：
- 如果候选人 > 10个：只匹配相似度最高的前10个
- 相似度计算：GPS距离 + 时间差 + 标签相似度综合评分

**发布后的展示**：
- ✅ 完全匿名（不显示用户名、头像、ID）
- ✅ 显示内容：
  - 时间（错过的时间）
  - 地点：显示 `address`（标准地址），不显示 `placeName`（私人标记）
  - 场所类型：显示 `placeType` 图标（如果有）
  - 描述（文字描述）
  - 标签名称 + 标签备注（让被错过的人能认出自己）
  - 状态（错过/再遇/邂逅等）
- ❌ 不显示内容：
  - GPS 精确坐标（保护隐私）
  - `placeName`（用户的私人标记，别人看不懂）
- ❌ 无法评论
- ❌ 无法点赞
- ❌ 无法私信（除非通过匹配验证）
- ❌ 无法联系发布者

**为什么社区要显示标签备注？**
- 让信息足够详细，被错过的人才能认出"这是在说我"
- 如果只显示标签名称（如"长发""戴眼镜"），太泛了，很多人都符合
- 但如果显示备注（如"圆框眼镜，很文艺"），就能精确识别
- 这是社区的核心价值：让被错过的人有机会看到并认出自己

**为什么社区不显示 `placeName`？**
- `placeName` 是用户的私人标记，如"常去的那家咖啡馆"
- 这对用户自己有意义（方便回忆），但对别人毫无意义
- 别人根本不知道"常去的那家咖啡馆"是哪家
- 社区需要标准地址（`address`），方便他人理解和确认地点
- 例如："北京市朝阳区建国门外大街1号" 比 "常去的那家咖啡馆" 更有用

#### 6.7 内容管理与反作弊

**无人工审核**：
- 项目为个人开发，无审核团队
- 依赖用户自律和社区规范

**无技术过滤**：
- 不对任何内容进行自动检测
- 私人记录：用户想写什么就写什么
- 发布到社区：用户自己负责，不留隐私信息

**发布到社区时的提醒**：

│                         │

│ 但无法修改。             │
│                         │
│ [我知道了，继续发布]    │
│ [取消]                  │
└─────────────────────────┘
```

**发布后的管理**：
- ✅ 用户可以删除自己发布到社区的内容
- ❌ 用户无法修改已发布的内容（防止发布后改成广告）
- ✅ 删除后，该记录仍然保留在用户的私人记录中
- ✅ 删除操作立即生效，其他用户无法再看到

**社区发布 = 快照**：
- 发布到社区时，会创建当前记录的快照
- 之后修改私人记录，不会同步到社区
- 如果想更新社区内容：先删除社区发布，再重新发布
- 这样保持社区内容的真实性，防止滥用

**为什么不做技术检测？**
- 技术检测容易误判，影响用户体验
- 依赖用户自律和社区氛围
- 社区本身是只读的，无法互动，即使有联系方式也无法直接联系

**为什么不提供举报功能？**
- 详见"关于页面"中的说明

**匹配反作弊机制**：

```
1. GPS 真实性验证
   - 检测 GPS 坐标是否合理
   - 检测是否突然跳跃
   - 检测是否为模拟器

2. 异常行为模式检测
   - 短时间内在不同地点创建大量记录（1小时内不同地点>5条）
   - 长期高频创建记录（每天>10条，连续7天）
   - 防止广撒网钓鱼

3. 双向验证（核心）
   - 通过标签判断身份
   - 骗子无法通过验证

4. 信用评分系统
   ✅ 应该扣分的情况：
   - GPS 异常（模拟器、跳跃）→ 信用分 -30
   - 异常行为模式（广撒网、高频滥用）→ 信用分 -10
   
   ❌ 不应该扣分的情况：
   - 验证时点击"不是，认错人了" → 不扣分
     （这是事实陈述，不是作弊行为）
   - 验证成功后点击"不，让它留在记忆里" → 不扣分
     （这是个人选择，不是作弊行为）
   - 正常频率的记录创建 → 不扣分
     （免费版支持无限记录，不限制正常使用）
   - 记录或社区发布中的任何内容 → 不扣分
     （不做内容检测，依赖用户自律）
   
   ⚠️ 信用分 < 0 → 禁止匹配功能

5. 实名认证（可选）
   - 手机号验证
   - 提高作弊成本
```

#### 6.8 GPS 异常检测详细机制

**检测目的**：
- 防止用户使用虚假定位参与匹配
- 防止骗子伪造 GPS 坐标广撒网
- 保证匹配的真实性

**触发时机**：
- 用户创建记录时（实时检测）
- 后台匹配时（二次验证）

**检测规则**：

**规则1：坐标合理性检测**
```
触发条件：
- GPS 坐标不在地球范围内
- 纬度 < -90° 或 > 90°
- 经度 < -180° 或 > 180°

判定：GPS 异常
扣分：-30分
```

**规则2：突然跳跃检测**
```
触发条件：
- 用户在短时间内创建了2条记录
- 两条记录的 GPS 距离 > 10公里
- 时间间隔 < 30分钟

例如：
- 10:00 在北京国贸站创建记录
- 10:20 在上海人民广场创建记录
- 距离：1200公里，时间：20分钟
- 判定：不可能，GPS 异常

判定：GPS 异常
扣分：-30分
```

**规则3：模拟器检测**
```
触发条件：
- GPS 坐标精度异常（accuracy = 0 或 accuracy > 1000米）
- GPS 坐标完全相同（连续多条记录的坐标一模一样，精确到小数点后6位）
- GPS 坐标呈现规律性（如每次都是整数坐标）
- 设备信息异常（检测到模拟器特征）

判定：GPS 异常
扣分：-30分
```

**规则4：异常行为模式检测**

**模式B：不同地点高频记录**
```
触发条件：
- 用户在短时间内（1小时内）
- 在不同地点（每个地点距离 > 1公里）
- 创建了 5条以上记录

例如：
- 18:00 在国贸站创建记录
- 18:15 在三里屯创建记录
- 18:30 在西单创建记录
- 18:45 在王府井创建记录
- 19:00 在天安门创建记录
- 判定：不合理，可能在用虚假定位广撒网

判定：异常行为
扣分：-10分
```

**模式C：长期高频记录**
```
触发条件：
- 用户每天创建 10条以上记录
- 连续 7天

判定：可能在滥用功能
扣分：-10分

说明：
- 正常用户偶尔一天创建10条记录是合理的
  （比如周末出去玩，遇到很多人）
- 但连续7天每天都创建10条以上，就不太正常了
```

**规则5：速度检测**
```
触发条件：
- 根据两条记录的 GPS 距离和时间间隔
- 计算移动速度
- 速度 > 120km/h（排除高铁、飞机等合理场景）

例如：
- 10:00 在北京国贸站创建记录
- 11:00 在天津站创建记录
- 距离：120公里，时间：1小时
- 速度：120km/h
- 判定：可能是高铁，不算异常

但如果：
- 10:00 在北京国贸站创建记录
- 10:30 在天津站创建记录
- 距离：120公里，时间：30分钟
- 速度：240km/h
- 判定：不合理（高铁也没这么快），GPS 异常

判定：GPS 异常
扣分：-30分
```

**规则6：室内/室外检测**
```
触发条件：
- GPS 信号强度异常
- 在室内（地铁、商场）GPS 精度应该较低
- 如果在地铁内 GPS 精度 < 10米，可疑

判定：可疑，但不直接扣分
标记：需要人工复核（如果将来有人工审核）
```

**检测流程**：

```
用户创建记录
  ↓
获取 GPS 坐标
  ↓
【实时检测】
  ├─ 坐标合理性检测
  ├─ 突然跳跃检测
  ├─ 模拟器检测
  ├─ 异常行为模式检测（不同地点高频、长期高频）
  ├─ 速度检测
  └─ 室内/室外检测
  ↓
判定结果：
  ├─ 正常 → 保存记录，参与匹配
  ├─ GPS异常 → 保存记录，但不参与匹配，扣信用分 -30
  ├─ 行为异常 → 保存记录，但不参与匹配，扣信用分 -10
  └─ 可疑 → 保存记录，标记为可疑，降低匹配优先级
```

## 🎨 UI/UX 设计原则

### 设计理念
**避免通用 AI 美学，打造独特的情感化设计**

### 视觉风格
- **色彩**：温暖、柔和、有情感张力
  - 避免：Inter、Roboto、Arial 等通用字体
  - 避免：紫色渐变等俗套配色
- **字体**：选择独特且优雅的字体
  - 英文：Playfair Display、Crimson Text、Libre Baskerville
  - 中文：思源宋体、霞鹜文楷、站酷高端黑
- **动画**：流畅、有意义的过渡动画
  - 状态升级动画
  - 页面切换动画
  - 微交互动画

### 情感化设计

**色彩应用规则**：
- **整体界面**：使用用户选择的主题（浅色/深色/跟随系统/朦胧/深夜/温暖/秋日）
- **状态卡片**：根据记录状态动态应用情感色调（叠加在主题之上）
- **强调色**：用于按钮、链接、重要提示（会员可自定义）

**不同状态的视觉语言**：
- **错过** 🌫️：朦胧、柔和、灰蓝色调
- **再遇** 🌟：明亮、惊喜、金色点缀
- **邂逅** 💫：温暖、激动、粉橙色调
- **重逢** 💝：圆满、幸福、玫瑰金色调
- **别离** 🥀：淡然、接受、玫瑰灰色调
- **失联** 🍂：平静、释怀、秋叶色调

**说明**：
- 状态色调是**情感氛围**，不是整体主题
- 例如：用户选择"深色"主题 + 记录状态是"错过" = 深色界面 + 灰蓝色调的卡片
- 例如：用户选择"温暖"主题 + 记录状态是"邂逅" = 米黄色界面 + 粉橙色调的卡片

### 交互原则
- **快速记录**：4步内完成记录（时间 → 地点 → 状态 → 保存）
- **无压力**：除了时间、地点、状态，其他字段全部可选
- **有温度**：文案温暖、鼓励、不说教
- **有仪式感**：重要时刻配合动画和音效

```dart
class EncounterRecord {
  String id;                    // 唯一标识
  DateTime timestamp;           // 时间戳
  Location location;            // 地点
  String? description;          // 描述（可选，最多500字）
  List<TagWithNote> tags;       // 标签 + 备注
  EmotionIntensity? emotion;    // 情绪强度（可选）
  EncounterStatus status;       // 当前状态（可修改）
  String? storyLineId;          // 所属故事线ID（可选）
  String? ifReencounter;        // "如果再遇"备忘

  String? backgroundMusic;      // 背景音乐
  WeatherInfo? weather;         // 天气信息
  DateTime createdAt;           // 创建时间
  DateTime updatedAt;           // 更新时间
}
```

**说明**：
- 每次见面 = 一个新记录
- 每个记录有自己的时间、地点、描述、状态
- 通过 `storyLineId` 把同一个人的多条记录关联起来
- `status` 可以修改（防止用户手滑点错）
- `description` 可选（有些错过无法用语言描述，最多500字）
- `emotion` 可选（情绪强度）
- `conversationStarter` 仅在状态为"邂逅"时使用，记录对话的契机（可选，最多500字）

```dart
class Match {
  String id;                    // 匹配ID
  String userAId;               // 用户A的ID
  String recordAId;             // 用户A的记录ID
  
  // 候选记录列表（可能来自不同用户）
  List<CandidateRecord> candidateRecords;
  
  // 用户A的验证
  String? userASelectedRecordId;     // 用户A选择的记录ID
  VerificationChoice? userAChoice;   // 用户A的选择
  
  // 对方用户的验证（动态确定是谁）
  String? otherUserId;               // 对方用户ID（用户A选择后确定）
  String? otherUserSelectedRecordId; // 对方用户选择的记录ID
  VerificationChoice? otherUserChoice; // 对方用户的选择
  
  MatchStatus status;           // 匹配状态
  MatchConfidence confidence;   // 匹配置信度（high/medium/low）
  double matchScore;            // 匹配评分（0-100，最高分记录的评分）
  DateTime matchedAt;           // 匹配时间
  DateTime? notifiedAt;         // 通知时间（6小时冷静期后）
  bool isPermanentlyKeptInMemory;  // 是否永久留在记忆里（不再匹配）
  DateTime? verifiedAt;         // 验证完成时间

**为什么社区要显示标签备注？**
- 让信息足够详细，被错过的人才能认出"这是在说我"
- 如果只显示标签名称（如"长发""戴眼镜"），太泛了，很多人都符合
- 但如果显示备注（如"圆框眼镜，很文艺"），就能精确识别
- 四重防御机制（GPS + 时间 + 标签验证）已经足够防止骗子

**地点显示优先级**：
1. 优先显示：`placeType` + `address`（最标准）
2. 其次显示：`address`（标准地址）
3. 再次显示：`placeName`（用户输入，无 GPS 时）
4. 最后显示：`cityName` 或 "未知地点"

**地点显示优先级**（社区页面）：
```dart
/// 获取社区帖子的地点显示文本
String getCommunityPostLocation(CommunityPost post) {
  String result = '';
  
  // 第一部分：场所类型（如果有）
  if (post.placeType != null) {
    result = post.placeType!.label;  // 显示文字，如"地铁"、"咖啡馆"
  }
  
  // 第二部分：标准地址（如果有）
  if (post.address != null && post.address!.isNotEmpty) {
    if (result.isNotEmpty) {
      result += ' · ${post.address}';  // 场所类型 · 地址
    } else {
      result = post.address!;  // 只有地址
    }
  }
  
  // 第三部分：如果没有 GPS，尝试 placeName
  if (result.isEmpty && post.placeName != null && post.placeName!.isNotEmpty) {
    result = post.placeName!;  // 显示用户输入的地点名称
  }
  
  // 第四部分：如果都没有，尝试城市名称
  if (result.isEmpty && post.cityName != null && post.cityName!.isNotEmpty) {
    result = post.cityName!;
  }
  
  // 实在没有，显示默认文本
  if (result.isEmpty) {
    result = '未知地点';
  }
  
  return result;
}
```

- 只有地址：`北京市朝阳区建国门外大街1号`
- 只有场所类型：`地铁`

## 📝 已确定的设计决策

### 1. 社交功能：树洞 + 隐藏匹配 ✅

**决策**：树洞模式 + 后台静默匹配

**树洞功能**：
- 用户可以匿名发布记录到社区
- 其他用户可以浏览，但无法互动
- 没有评论、点赞功能

**匹配功能**：
- 用户知道有匹配功能（产品营销卖点）
- 但后台静默运行，用户不知道"正在匹配中"，也不知道"匹配失败"
- 只在成功时推送通知
- 成功率：0.01%
- 五重防御机制：
  1. 对方也使用 Serendipity
  2. 对方也记录了那次错过
  3. 双方都将记录加入了故事线（技术要求）
  4. GPS 距离 < 200米 + 时间差 < 15分钟 + 地点名称模糊匹配
  5. 标签的双向验证（防骗核心）

**匹配流程**：
- 匹配成功 → 等待6小时冷静期 → 推送通知
- 双向验证（通过标签判断身份）
- 双方都确认 → 解锁私信（7天有效期）
- 任何一方拒绝 → 匹配失败

**产品哲学**：
- 相信奇迹，但不强求
- 主要功能：情感记录（99.99%）

- 用户知道有匹配功能，但不知道"正在匹配中"
- 如果根本没匹配上，不会收到通知；进入验证后失败，会温柔告知
