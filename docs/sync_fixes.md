## Bug1 — signOut did not clear local data

**File**: `lib/core/providers/auth_provider.dart`

**Root cause**: `signOut()` only called `_invalidateDataProviders()` (clears memory cache).
Never called `storageService.clearUserData()`, so Hive retained all user A data.

**Impact**: When user B logs in, `bindOfflineDataToUser` only binds `ownerId==null` records.
User A records (`ownerId==userA`) stay in Hive forever, causing storage bloat and
potential data leakage if any filter has a bug.

**Fix**: Added `await storageService.clearUserData()` in `signOut()` before invalidating providers.
User data is re-downloaded from cloud on next login via `_triggerSync`.

```dart
// BEFORE
await _repository.signOut();
_invalidateDataProviders();
state = const AsyncValue.data(null);

// AFTER
await _repository.signOut();
final storageService = ref.read(storageServiceProvider);
await storageService.clearUserData(); // NEW: prevent cross-user data leakage
_invalidateDataProviders();
state = const AsyncValue.data(null);
```

## Bug2a — hasCheckedInToday ignored userId

**File**: `lib/core/repositories/check_in_repository.dart`

**Root cause**: Used fixed `todayId = today.millisecondsSinceEpoch.toString()` as Hive key lookup,
completely ignoring which user is logged in.

**Impact**: User A checks in today. User A logs out, user B logs in.
`hasCheckedInToday()` finds user A's record via the same fixed key and returns `true`.
User B cannot check in.

**Fix**: Added `{String? userId}` parameter. Now queries `getCheckInsByUser(userId)`
and checks if any record matches today's date.

```dart
// BEFORE
bool hasCheckedInToday() {
  final todayId = _getTodayDate().millisecondsSinceEpoch.toString();
  return _storageService.getCheckIn(todayId) != null;
}

// AFTER
bool hasCheckedInToday({String? userId}) {
  final today = _getTodayDate();
  return _storageService.getCheckInsByUser(userId).any((c) => c.date == today);
}
```

## Bug2b — checkIn() used fixed todayId for dedup

**File**: `lib/core/repositories/check_in_repository.dart`

**Root cause**: `checkIn()` called `getCheckIn(todayId)` with a fixed key that did not
match the actual ID generated by `CheckInRecord.create()` (which uses a random timestamp).
Also did not filter by userId.

**Fix**: Replaced the fixed-key lookup with a call to the fixed `hasCheckedInToday(userId: userId)`.

```dart
// BEFORE
final todayId = _getTodayDate().millisecondsSinceEpoch.toString();
final existing = _storageService.getCheckIn(todayId);
if (existing != null) throw StateError('Already checked in today');

// AFTER
if (hasCheckedInToday(userId: userId)) {
  throw StateError('Already checked in today');
}
```

## Bug2c — Stats methods did not filter by userId

**Files**: `check_in_repository.dart`, `check_in_provider.dart`

**Root cause**: `calculateConsecutiveDays()`, `getTotalCheckInDays()`,
`getCurrentMonthCheckInDays()`, `getCheckInDatesInMonth()` all called `getAllCheckIns()`
which returns records from every user stored in Hive.

**Impact**: On a shared device, streak/total/monthly stats include other users' check-ins.

**Fix**:
1. All repository stats methods now accept `{String? userId}` and use `getCheckInsByUser(userId)`
2. `CheckInNotifier._refresh({String? userId})` passes userId to all stats calls
3. Public `refresh()` reads current userId from `authProvider` automatically
4. `getCheckInDatesInMonth` and `resetAllCheckIns` also accept userId

## Issue3 — authStateChanges polled every 5 seconds

**File**: `lib/core/repositories/custom_server_auth_repository.dart`

**Root cause**: `Stream.periodic(Duration(seconds: 5))` fired a `GET /auth/me` every 5s.
`AuthNotifier` (a `StreamNotifier`) listens to this stream, so every tick could
trigger a state update and cause all `watch(authProvider)` widgets to rebuild.

**Fix**: Changed to `Duration(seconds: 300)`. Token validity is maintained by
`HttpClientService`'s two-layer auto-refresh (proactive pre-expiry refresh + 401 retry).
Application-level polling at 5s is unnecessary.

## Issue4 — StoryLinesProvider rethrow blocked UI refresh

**File**: `lib/core/providers/story_lines_provider.dart`

**Root cause**: `createStoryLine`, `updateStoryLine`, `deleteStoryLine` called `rethrow`
immediately on cloud sync failure, before `refresh()` could execute.
Local write succeeded but UI never showed the result.
Inconsistent with `RecordsProvider` which swallows cloud errors and always refreshes.

**Fix**: Save the exception reference first, run `refresh()`, then throw at the end.

```dart
// BEFORE
try {
  await _syncService.uploadStoryLine(currentUser, storyLine);
} catch (e) {
  rethrow; // refresh() never runs
}
await refresh();

// AFTER
Exception? syncException;
try {
  await _syncService.uploadStoryLine(currentUser, storyLine);
} catch (e) {
  syncException = e is Exception ? e : Exception(e.toString());
}
await refresh(); // always runs regardless of cloud result
if (syncException != null) throw syncException; // throw after UI is updated
```

Same pattern applied to `updateStoryLine` and `deleteStoryLine`.

## Issue5 — Full sync did not delete locally-orphaned records

**File**: `lib/core/services/sync_service.dart`

**Root cause**: `_downloadRemoteData()` only merged (write if cloud record exists locally or is newer).
Deleted records are absent from the API response, so they were never removed from local storage.
Device B would never see deletions made on device A.

**Fix**: When `lastSyncTime == null` (full sync), compare local vs remote ID sets and
delete any local record not present in the cloud response. Applied to records, story lines,
and check-ins.

```dart
if (isFullSync) {
  final remoteIds = remoteRecords.map((r) => r.id).toSet();
  for (final local in _storageService.getRecordsByUser(user.id)) {
    if (!remoteIds.contains(local.id)) {
      await _storageService.deleteRecord(local.id);
    }
  }
}
```

**Note**: Incremental sync still cannot propagate deletes (requires a server-side
deletion tombstone table). Full sync serves as the fallback.

## Issue6 — SyncStatusNotifier Timer could not be cancelled

**File**: `lib/core/providers/sync_status_provider.dart`

**Root cause**: `syncSuccess()` and `syncError()` used bare `Future.delayed` with no
Timer reference stored. `StateNotifier.dispose()` had no way to cancel them.

**Fix**:
- Added `Timer? _autoResetTimer` field
- Cancel existing timer before creating a new one (prevents overlapping timers)
- Override `dispose()` to cancel the timer
- Added `import 'dart:async'` at top of file

```dart
// BEFORE
Future.delayed(const Duration(seconds: 3), () {
  if (mounted && state.status == SyncStatus.success) {
    state = state.copyWith(status: SyncStatus.idle);
  }
});

// AFTER
_autoResetTimer?.cancel();
_autoResetTimer = Timer(const Duration(seconds: 3), () {
  if (mounted && state.status == SyncStatus.success) {
    state = state.copyWith(status: SyncStatus.idle);
  }
});

@override
void dispose() {
  _autoResetTimer?.cancel();
  super.dispose();
}
```
