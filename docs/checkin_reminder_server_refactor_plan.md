# 签到与签到提醒服务端化重构计划

## 文档目的

这份文档用于记录“签到系统从本地主导迁移为服务端权威源，以及签到提醒从纯本地通知升级为混合方案”的重构计划，避免后续实现过程中上下文丢失、方向漂移或重复讨论。

---

## 当前结论

### 现状归类
当前签到提醒方案属于：

- **纯本地定时通知**

### 当前实现特点
- Flutter 端使用 `flutter_local_notifications`
- 通过 `zonedSchedule(...)` 本地调度每日提醒
- 提醒文案在**调度时**生成，而不是触发时生成
- 没有服务端到点推送
- 没有显式的“前台补偿”机制

### 当前问题
- Android 后台、锁屏、待机时提醒不稳定
- 某些情况下会出现“切回前台才看到提醒”
- 纯本地方案受系统电池优化、后台限制、厂商策略影响太大
- 用户对“签到提醒”天然预期是**到点就该提醒**，但当前方案无法稳定满足
- 当前签到数据属于**本地主读、云端同步副本**，不适合多设备长期演进
- 同一账号在多设备下，签到状态、连续天数、提醒文案可能出现不一致
- 如果未来提醒改由服务端推送，而签到仍以本地为准，服务端将无法可靠判断“今天是否已签到”

---

## 重构目标

将签到系统与签到提醒一起升级为：

- **签到数据：服务端权威源，本地缓存与离线副本**
- **签到提醒：本地提醒 + 服务端推送兜底** 的混合方案

目标不是复制大厂全套推送体系，而是做一个**独立开发者可维护、架构清晰、体验显著提升**的 MVP。

---

## 最终目标形态

### 目标方案分类
- **签到：服务端权威源 + 本地缓存**
- **提醒：混合方案**

### 责任划分

#### 1. Flutter 本地层
负责：
- 未登录用户的签到与提醒能力
- 登录用户的签到缓存与 UI 快速展示
- 离线场景的基础提醒
- 服务端不可用时的 fallback

#### 2. 服务端签到能力
负责：
- 登录用户的签到真相
- 今日是否已签到判断
- 连续签到天数、累计签到天数等核心统计
- 多设备下签到状态一致性

#### 3. 服务端推送
负责：
- 登录用户的正式签到提醒
- 后台 / 锁屏 / 待机场景下更可靠的到点提醒
- 多设备下更一致的提醒策略

#### 4. 业务规则统一由服务端判定
服务端负责判断：
- 当前是否开启签到提醒
- 当前是否已经签到
- 是否到达用户设定时间
- 今天是否已经发过提醒

---

## 产品策略

### 为什么要做
- 签到提醒属于“承诺型功能”
- 用户不关心系统限制，只关心“为什么没提醒”
- 当前真实测试已经证明：纯本地方案在正式后台场景不可靠
- app 已明确支持多设备，签到继续本地主导会让多设备一致性越来越差
- 只要未来提醒走服务端推送，签到状态也必须提升为服务端可可靠判定的权威数据

### 为什么不直接全量重做
- 独立开发者不适合一步到位做大厂级推送基建
- 厂商通道全适配、复杂路由、消息中心、运营推送都不是当前重点
- 当前最值钱的目标是：**先把签到提醒做可靠**

---

## 重构边界

### 本次要做
- 针对**签到数据权威源迁移**与**签到提醒服务端化**
- 只做**服务端推送 MVP**
- 先支持**登录用户**
- 本地提醒继续保留
- 本地签到能力继续保留给未登录用户
- 推送只解决“按时提醒”问题，不扩展成通用通知平台

### 本次不做
- 不做全站通知中心
- 不做聊天消息推送体系
- 不做运营推送后台
- 不做所有通知统一服务化
- 不做国内安卓厂商通道全家桶适配
- 不做复杂 AB 实验与送达率面板

---

## 推荐落地方案

## Phase 1：梳理并稳定现有本地签到与本地提醒

目标：
- 让当前本地签到与本地提醒逻辑尽可能稳定、可解释、可维护

任务：
- 明确当前签到数据读取与同步链路
- 明确当前签到提醒的触发链路
- 修正文案生成时机问题
- 修正“提醒文案与当前签到状态不一致”的问题
- 保留本地提醒作为 fallback
- 保持关于页 FAQ 与真实能力一致

产出：
- 本地签到从“真相源”降级为“缓存 / 离线能力”候选
- 本地提醒成为“基础能力”而不是唯一依赖

---

## Phase 2：签到权威源迁移到服务端

目标：
- 让登录用户的签到状态、连续天数、今日是否签到等核心判断都以服务端为准

核心能力：
- 服务端保存签到记录并作为最终真相
- 登录用户签到时优先写服务端，再回写本地缓存
- Flutter 端优先从服务端读取签到状态，本地只做缓存展示
- 多设备下统一签到状态与连续签到计算
- 明确离线场景的降级策略

MVP 范围建议：
- 未登录用户继续保留本地签到
- 登录用户逐步切到服务端主写 / 主读
- 先不做复杂离线冲突合并

---

## Phase 3：服务端推送 MVP

目标：
- 为登录用户提供真正可用的后台签到提醒

核心能力：
- 设备推送 token 上报
- 服务端保存提醒设置
- 服务端定时扫描“该提醒的用户”
- 服务端基于服务端签到数据判断今日是否已签到
- 满足条件时发送推送
- 防止同一用户一天重复推送

MVP 范围建议：
- Android 走 FCM
- iOS 走 APNs
- 只处理签到提醒
- 只处理正式提醒，不做测试消息后台化

---

## Phase 4：混合策略收口

目标：
- 明确本地提醒与服务端推送之间的职责边界，避免双推送或状态错乱

建议策略：
- 未登录用户：只走本地提醒
- 登录用户：优先服务端推送，本地提醒作为 fallback
- 如果后续双端重复提醒问题明显，可进一步收紧策略为：
  - 登录后仅服务端推送
  - 登出或离线数据场景才用本地提醒

这个策略到实现阶段再定，不要现在过早定死。

---

## 架构原则

### 1. 不写向后兼容垃圾逻辑
项目仍在开发期，直接按目标架构推进，不做多余兼容层。

### 2. 服务端负责“签到真相与是否该发”
不要让 Flutter 和服务端各自独立判断，否则一定会出现分叉。

### 3. Flutter 负责“设置、缓存与设备注册”
客户端只负责：
- 用户设置提醒时间
- 开关提醒
- 获取并上传 push token
- 展示推送权限状态
- 缓存签到状态与最近签到记录

### 4. 去重必须服务端兜底
必须能回答：
- 今天给这个用户发过没有？
- 这个设备 token 还有效吗？
- 这个用户是否已经签到？

### 5. 签到权威源必须唯一
登录用户的签到真相只能由服务端拍板，不能继续让本地与服务端并列为两个来源。

### 6. 不把签到提醒抽象成通用通知平台
当前只做签到提醒，不提前抽象成“万能通知系统”。

---

## 需要新增的核心数据

以下是方向，不是最终字段定稿。

### Flutter 侧
用户设置中可能需要补充：
- 是否允许服务端签到提醒
- 提醒时区
- 设备 push token 上报状态
- 最近一次 token 同步时间
- 登录用户签到缓存的最近同步时间

### 服务端侧
至少需要有能力存储：

#### 签到相关
- 用户 ID
- 签到日期
- 签到时间戳
- 连续签到天数（可存可算，最终以服务端规则为准）
- 累计签到天数（可存可算）

#### 推送相关
- 用户 ID
- 平台类型（Android / iOS）
- push token
- 提醒开关
- 提醒时间
- 用户时区
- token 状态（有效 / 失效）
- 最近一次发送日期

---

## 服务端判断逻辑（目标）

### A. 签到判定

1. 登录用户签到时，请求服务端创建签到记录
2. 服务端判断今天是否已经签到
3. 若未签到，则落库并返回最新签到状态
4. Flutter 收到结果后回写本地缓存
5. 多设备统一读取服务端结果

### B. 提醒判定

1. 找出所有开启签到提醒的登录用户
2. 按用户时区判断是否到达提醒时间窗口
3. 判断该用户今天是否已经签到
4. 判断今天是否已经发过提醒
5. 若未签到且未发送过，则发推送
6. 记录发送结果
7. 对失效 token 做降级处理

---

## 风险点

### 1. 双推送
如果本地与服务端同时生效，用户可能一天收到两条。

### 2. 时区处理
签到提醒属于“按用户本地日期触发”的功能，时区不能乱。

### 3. 多设备问题
一个账号多个设备时，要提前决定：
- 是否所有设备都推
- 还是只推最近活跃设备

MVP 可以先简单处理，但必须提前意识到这个问题。

### 4. token 失效
卸载重装、系统更新、换机后 token 会变化，必须有失效回收机制。

### 5. 离线签到冲突
如果登录用户离线时仍允许先本地签到，后续联网上传时必须定义冲突规则，否则服务端真相会被污染。

---

## 设计决策（当前已确认）

以下决策已在当前讨论中确认，后续实现直接按此执行，不再重复悬而未决：

1. **登录用户不再保留本地提醒**
   - 只走服务端推送
   - 目的：避免双提醒、双链路、双真相

2. **一个账号多设备时，允许多端都收到签到提醒**
   - 先采用最直接、最符合多设备登录直觉的策略
   - 当前不引入“主设备 / 最近活跃设备”仲裁逻辑

3. **服务端推送失败时，不做本地提醒兜底**
   - 项目仍在开发期，先保持单链路清晰
   - 不引入双系统补偿逻辑，避免复杂度失控

4. **时区以服务端持久化的用户时区为准**
   - 第一版由设备上报时区并保存到服务端
   - 服务端调度使用持久化时区，避免完全依赖设备瞬时漂移

5. **未登录用户继续只保留本地提醒与本地签到**
   - 未登录用户没有账号、push token 和服务端用户语义
   - 因此不接入服务端推送链路

6. **登录用户必须在线签到**
   - 不允许先本地签到、后续再补传
   - 这样可以保持服务端为唯一签到真相源，避免离线冲突合并

7. **测试推送功能仅保留在本地开发工具内**
   - 不暴露到正式用户界面
   - 先聚焦正式签到提醒链路，不扩展调试入口

## Phase 3 实现边界（已冻结）

### Phase 3 要做的
- 服务端保存设备 push token
- 登录 / 启动时客户端上报 token
- 服务端按用户时区和提醒时间筛选待提醒用户
- 服务端基于服务端签到数据判断今日是否已签到
- 通过 FCM / APNs 发送签到提醒
- 记录发送结果并回收失效 token
- 未登录用户继续保留本地提醒，不接入服务端推送

### Phase 3 不做的
- 向后兼容旧的“登录用户本地提醒 + 本地签到补传”
- 推送失败后再由本地提醒兜底补提醒
- 主设备 / 副设备复杂仲裁
- 离线签到冲突合并策略
- 面向正式用户的测试推送后台入口

## 建议实施顺序

### 第一步
- 先把当前本地签到与本地提醒问题彻底厘清
- 尤其是签到主读链路、文案生成时机和触发可靠性问题

### 第二步
- 设计服务端签到数据模型与提醒接口
- 不急着写代码，先把链路画清楚

### 第三步
- Flutter 接入 push token 获取与上报
- 同时梳理登录用户签到缓存策略

### 第四步
- 服务端先实现“签到权威源”能力
- 再实现签到提醒调度任务

### 第五步
- 完成 Flutter / Server 联调
- 验证多设备签到状态一致性

### 第六步
- 最后再决定登录用户的本地提醒是否保留

---

## 验收标准（MVP）

达到以下标准，才算这次重构真正成立：

- 登录用户在正常联网情况下，签到状态以服务端为准
- 同一账号在多设备上看到一致的今日签到状态与连续签到天数
- 登录用户在正常联网情况下，后台/锁屏场景能稳定收到签到提醒
- 服务端不会在用户已签到后继续发提醒
- 同一用户同一天不会被重复提醒
- 未登录用户仍可继续使用本地签到与本地签到提醒
- 设置修改后，服务端提醒时间能及时更新
- 代码职责清晰：客户端管设置与缓存，服务端管签到真相与提醒触发

---

## 当前阶段结论

这不是一个“要不要推翻重写”的问题，而是一个“要不要把签到系统与签到提醒一起升级为正式能力”的问题。

结论已经明确：

- **方向上要把签到权威源迁移到服务端**
- **方向上要把签到提醒升级为混合方案**
- **实现上先做签到服务端化 + 提醒服务端化 MVP**
- **不要一开始做成大厂级通知平台**
