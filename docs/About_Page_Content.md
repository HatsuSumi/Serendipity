## 双页信息架构说明

### 页面结构
本次采用双页结构：

1. `关于 Serendipity`
2. `设计决策`

### 页面入口关系
- 设置页 / 我的页 → `关于 Serendipity`
- 在 `关于 Serendipity` 页底部或中段提供入口：
  - `查看更多设计决策`
  - `为什么产品这样设计？`
- 点击后进入 `设计决策` 页面

### 文案归属规则
以下文案内容保持原文不变，仅在实现时按页面拆分：

#### 归属 `关于 Serendipity` 页面
- 关于 Serendipity
- 状态说明
- 如何记录多次见面？
- 设计理念
- 技术选择
- 开发者

#### 归属 `设计决策` 页面
- 为什么树洞不能互动？
- 记录和社区帖子的关系
- 为什么不提供举报功能？
- 为什么有时候设置了页面切换动画，进入时没有，反而退出时有？
- 为什么签到系统没有补签功能？
- 为什么有时候登出时会看不到之前创建的记录/故事线？
- 为什么不提供地图视图查看所有记录的点位？
- 为什么“手机号/邮箱注册”都不支持发送验证码？
- 统计页的标签词云图是怎么实现的？
- 既然技术栈用了 Flutter，为什么只做了 Android 和 iOS 呢？

### 实现原则
- 不改动现有文案内容
- 只调整页面承载方式与入口关系
- 文案顺序可按页面拆分后分别组织

│                         │

│ 符合产品调性。           │
│                         │
│ 这个 app 叫              │

│ 2. 未登录时创建2条新记录 │
│ 3. 登录账号B（不是账号A）│
│ 4. 自动绑定：2条新记录   │
│    归属账号B             │
│                         │
│ 核心原则：               │
│ "谁先登录，               │
│  离线数据就归谁"         │
│                         │
│ 为什么这样设计？         │
│                         │
│ 第一个原因：             │
│ 支持离线使用。           │
│                         │
│ 你可以在未登录时         │
│ 创建记录，               │
│ 登录后自动归属到         │
│ 你的账号。               │
│                         │
│ 第二个原因：             │
│ 数据隔离更安全。         │
│                         │
│ 每个账号只能看到         │
│ 自己的数据，             │
│ 不会混淆。               │
│                         │
│ 第三个原因：             │
│ 符合直觉。               │
│                         │
│ 你创建数据后登录，       │
│ 数据自然归属该账号。     │
│                         │
│ 如何避免数据丢失？       │
│                         │
│ 1. 创建数据后及时登录    │
│ 2. 不要频繁切换账号      │
│ 3. 记住哪些数据归属      │
│    哪个账号               │
│                         │
│ 如果找不到数据了？       │
│                         │
│ 数据一定在某个账号里，   │
│ 尝试登录所有账号查看。   │
│                         │
│ 注意：                   │
│ 数据一旦绑定到账号，     │
│ 就无法在登出状态下查看。 │
│                         │
│ 这不是 bug，             │
│ 而是多账号数据隔离的     │
│ 必然结果。               │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 为什么不提供地图视图     │
│ 查看所有记录的点位？     │
│                         │
│ 这个 app 没有地图视图    │
│ 来展示所有记录的位置。   │
│                         │
│ 为什么？                 │
│                         │
│ 第一个原因：             │
│ 高德地图 Flutter 插件    │
│ 与当前 Flutter 版本      │
│ 不兼容。                 │
│                         │
│ amap_flutter_map 和      │
│ amap_flutter_base 包     │
│ 使用了 Dart 2.x 时代的   │
│ hashValues() 方法，       │
│ 该方法在 Dart 3.x 中     │
│ 已被移除。               │
│                         │
│ 所有版本的高德地图       │
│ Flutter 插件都未更新     │
│ 以支持 Dart 3.x，        │
│ 导致无法编译。           │
│                         │
│ 第二个原因：             │
│ 替代方案已经足够。       │
│                         │
│ 虽然没有地图视图，       │
│ 但你可以：               │
│                         │
│ 1. 在创建记录时          │
│    自动获取 GPS 定位     │
│ 2. 手动输入地点名称      │
│ 3. 使用地点历史记录      │
│    快速选择              │
│ 4. 在编辑模式下          │
│    重新定位或清除 GPS    │
│                         │
│ 这些功能已经能够满足     │
│ 大多数用户的需求。       │
│                         │
│ 第三个原因：             │
│ 一人开发的现实限制。     │
│                         │
│ 等待高德地图插件更新     │
│ 遥遥无期，而且即使更新   │
│ 也不一定会支持 Dart 3.x。│
│                         │
│ 自己实现地图功能         │
│ 需要大量时间和精力，     │
│ 对于一人项目来说         │
│ 不现实。                 │
│                         │
│ 所以选择了接受这个       │
│ 技术限制，并提供了       │
│ 最优的替代方案。         │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 为什么“手机号/邮箱注册”  │
│ 都不支持发送验证码？     │
│                         │
│ 这个 app 目前的注册和    │
│ 登录流程，不提供短信     │
│ 验证码，也不提供邮箱     │
│ 验证码。                 │
│                         │
│ 为什么？                 │
│                         │
│ 第一个原因：             │
│ 验证码服务都依赖         │
│ 第三方通道。             │
│                         │
│ 手机短信验证码需要       │
│ 接入短信服务商，         │
│ 通常还要求企业资质。     │
│                         │
│ 邮箱验证码虽然门槛       │
│ 比短信低，但依然需要     │
│ 稳定的邮件发送服务、     │
│ 投递率维护和风控处理。   │
│                         │
│ 第二个原因：             │
│ 持续成本和维护成本高。   │
│                         │
│ 短信按条计费，           │
│ 邮件服务也不是零成本。   │
│                         │
│ 更重要的是，验证码系统   │
│ 不是接上就结束了，       │
│ 还要处理：               │
│                         │
│ 发送失败                 │
│ 延迟到达                 │
│ 频率限制                 │
│ 恶意刷接口               │
│ 邮件进入垃圾箱           │
│                         │
│ 对一人开发项目来说，     │
│ 这是一套长期维护负担。   │
│                         │
│ 第三个原因：             │
│ 现阶段没有这个必要。     │
│                         │
│ 对于这个 app 的使用场景， │
│ 账号系统更需要的是       │
│ 简单、稳定、可维护，     │
│ 而不是为了“更完整”       │
│ 强行接入验证码体系。     │
│                         │
│ 第四个原因：             │
│ 少一层验证码，           │
│ 不代表更不安全。         │
│                         │
│ 安全不只取决于           │
│ 有没有验证码，           │
│ 还取决于整体认证设计、   │
│ 密码存储、接口限流、     │
│ 设备管理和数据隔离。     │
│                         │
│ 所以现阶段选择的是：     │
│ 不做一个看起来完整、     │
│ 实际维护不起的           │
│ 验证码系统。             │
│                         │
│ 先把真正核心的体验       │
│ 做稳定，比形式上的       │
│ “支持验证码”更重要。     │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 统计页的标签词云图       │
│ 是怎么实现的？           │
│                         │
│ Flutter 最常用的图表库   │
│ fl_chart 不提供词云图。  │
│                         │
│ 所以这个词云图是         │
│ 从零手写的。             │
│                         │
│ 核心算法：               │
│ 阿基米德螺旋碰撞检测     │
│ 布局引擎                 │
│                         │
│ 简单来说：               │
│                         │
│ 1. 按词频降序排列，       │
│    高频词优先占据        │
│    画布中心              │
│                         │
│ 2. 每个词从中心出发，    │
│    沿螺旋线依次尝试      │
│    候选坐标              │
│                         │
│ 3. 每次候选都做碰撞      │
│    检测（AABB），        │
│    找到第一个不重叠的    │
│    位置就放置            │
│                         │
│ 4. 用 TextPainter        │
│    精确测量每个词的      │
│    渲染尺寸，避免        │
│    重叠或空间浪费        │
│                         │
│ 5. 25% 的词随机竖排，    │
│    模拟真实词云的        │
│    纵向穿插效果          │
│                         │
│ 6. 颜色按词频在          │
│    主色→辅色之间         │
│    线性渐变              │
│                         │
│ 感兴趣的话可以在         │
│ GitHub 上查看源代码：    │
│                         │
│ tag_cloud_card.dart      │
│ (serendipity_app/lib/    │

│  widgets/)              │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 既然技术栈用了 Flutter， │
│ 为什么只做了 Android    │
│ 和 iOS 呢？             │
│                         │
│ 因为这个 app 的核心场景  │
│ 是记录日常生活中         │
│ 擦肩而过的人。           │
│                         │
│ 这种瞬间往往来得很快，    │
│ 你需要立刻掏出设备记下。 │
│                         │
│ 而在这种场景里，         │
│ 掏出手机总是比           │
│ 掏出电脑更快。           │
│                         │
│ 但这还不只是            │
│ "手机和电脑"的区别。     │
│                         │
│ 这个 app 更需要的是      │
│ 随时、低阻力、稳定的     │
│ 原生移动端体验。         │
│                         │
│ Web 虽然也能在手机上用， │

│ 那么直接、顺手、稳定。   │
│                         │
│ Flutter 确实支持多端，   │
│ 但支持多端               │
│ 不代表要先做所有端。     │
│                         │
│ 所以即使用了 Flutter，   │
│ 现阶段也优先做           │
│ Android 和 iOS。        │
│                         │

│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
