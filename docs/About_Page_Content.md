# Serendipity - 关于页面内容

> 本文档包含 Serendipity app 中"关于页面"的完整文案内容

---

## 关于 Serendipity

```
┌─────────────────────────┐
│ 关于 Serendipity        │
├─────────────────────────┤
│                         │
│ Serendipity 是一个记录  │
│ "错过"的 app。           │
│                         │
│ 不是为了重逢，           │
│ 而是为了记住那些         │
│ 转瞬即逝的瞬间。         │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 状态说明                 │
│                         │
│ 这个 app 有七种状态，    │
│ 核心区别在于：           │
│ 有没有交流过。           │
│                         │
│ 🌫️ 错过                 │
│ 第一次看到，但没说话     │
│                         │
│ 🙈 回避                 │
│ 看到了，但刻意避开       │
│ （不敢看、低头、转身）   │
│                         │
│ 🌟 再遇                 │
│ 又看到了，但还是没说话   │
│ （命运的第二次机会）     │
│                         │
│ 💫 邂逅                 │
│ 终于说话了！             │
│ （第一次真正的见面）     │
│                         │
│ 💝 重逢                 │
│ 分开后再次相遇           │
│ （需先经历别离）         │
│                         │
│ 🥀 别离                 │
│ 主动结束了               │
│ （分手、好聚好散）       │
│                         │
│ 🍂 失联                 │
│ 被动消失，再也没见过     │
│                         │
│ 简单来说：               │
│ 没说过话 → 错过/回避/再遇 │
│ 说过话了 → 邂逅          │
│ 分开后又遇 → 重逢        │
│ 主动结束 → 别离          │
│ 被动消失 → 失联          │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 如何记录多次见面？       │
│                         │
│ 每次见面 = 一条新记录。  │
│                         │
│ 例如：                   │
│ 2月8日在地铁看到 TA      │
│   → 创建记录1，状态"错过"│
│                         │
│ 2月9日又看到 TA          │
│   → 创建记录2，状态"再遇"│
│                         │
│ 2月10日终于说话了        │
│   → 创建记录3，状态"邂逅"│
│                         │
│ 2月15日分手了            │
│   → 创建记录4，状态"别离"│
│                         │
│ 3月1日又遇到了           │
│   → 创建记录5，状态"重逢"│
│                         │
│ 然后通过"故事线"功能，   │
│ 把这5条记录关联起来，    │
│ 形成完整的时间线。       │
│                         │
│ 不要在旧记录上修改状态！ │
│ "更改状态"功能只是用来   │
│ 修正手滑点错的情况。     │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 设计理念                 │
│                         │
│ 这个 app 的侧重点是      │
│ 帮助你记录错过的瞬间，   │
│ 而不是邂逅或重逢。       │
│                         │
│ 邂逅后如果继续在一起，   │
│ 就不需要再记录了，       │
│ 因为已经不是"错过"，     │
│ 而是"拥有"。             │
│                         │
│ 只有分开了（别离/失联），│
│ 才又回到"错过"的状态。   │
│ 这时如果再次相遇，       │
│ 就是"重逢"。             │
│                         │
│ 这个 app 的核心价值是：  │
│ 记录、回忆、接受遗憾。   │
│                         │
│ 有些错过，               │
│ 注定只能被记住。         │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 为什么树洞不能互动？     │
│                         │
│ 这个 app 有一个"树洞"   │
│ 功能，你可以匿名发布     │
│ 你的错过。               │
│                         │
│ 但树洞没有评论、点赞、   │
│ 也无法联系发布者。       │
│                         │
│ 你可以在树洞中看到       │
│ 是否有别人记录了你。     │
│                         │
│ 为什么要这么"残酷"？     │
│                         │
│ 第一个原因：防骗。       │
│                         │
│ 如果树洞可以互动，       │
│ 骗子会怎么做？           │
│                         │
│ 1. 看到你的记录          │
│ 2. 假装就是那个人        │
│ 3. 主动联系你            │
│ 4. 骗取信任              │
│                         │
│ 零互动 = 骗子无法联系你。│
│                         │
│ 第二个原因：             │
│ 这才是"错过"的本质。     │
│                         │
│ 你发布到树洞后，         │
│ 也许 TA 会看到，         │
│ 也许 TA 会想'这不是我吗？'，       │
│ 但你永远不会知道。       │
│                         │
│ 这种"也许"的不确定性，   │
│ 才是最美的遗憾。         │
│                         │
│ 双方都发布到树洞，       │
│ 但是双方都取不到联系。   │
│                         │
│ 如果可以评论、点赞，   │
│ 那就不是"错过"了，       │
│ 而是"社交"。             │
│                         │
│ 第三个原因：             │
│ 这个 app 叫"错过了么"，  │
│ 不是"邂逅了么/重逢了么/  │
│ 失去了么"。              │
│                         │
│ 有些错过，               │
│ 注定只能是错过。         │
│                         │
│ 但至少，                 │
│ 我们记住了。             │
│                         │
│ 这是一种「刀子美学」设计——  │
│ 但这就是错过的本质——     │
│ 有些遗憾，               │
│ 注定无法弥补。           │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 记录和社区帖子的关系     │
│                         │
│ 社区帖子可以理解为       │
│ 记录的"快照"状态。       │
│                         │
│ 当你发布记录到社区时，   │
│ 会创建一个快照，         │
│ 包含以下字段：           │
│                         │
│ 错过时间                 │
│ 发布时间                 │
│ 地址                     │
│ 地点名称                 │
│ 场所类型                 │
│ 省市区                   │
│ 描述                     │
│ 标签                     │
│ 状态                     │
│                         │
│ 不包含以下字段：         │
│                         │
│ 精确GPS坐标              │
│ 情绪强度                 │
│ 对话契机                 │
│ 背景音乐                 │
│ 天气                     │
│ "如果再遇"备忘           │
│                         │
│ 发布后的规则：           │
│                         │
│ 1. 不能直接修改          │
│    发布后只能删除，       │
│    不能修改内容。         │
│                         │
│ 2. 可以重新发布          │
│    修改本地记录后，       │
│    可以重新发布，         │
│    会替换旧帖子。         │
│                         │
│ 3. 修改不显示的字段      │
│    如果你只修改了         │
│    "情绪强度"、"天气"等  │
│    不显示在社区的字段，   │
│    系统不会提示你重新发布，│
│    因为社区内容没有变化。 │
│                         │
│ 为什么这样设计？         │
│                         │
│ 保持真实性               │
│ 发布的那一刻就是         │
│ 你当时的真实感受         │
│                         │
│ 防止滥用                 │
│ 不能修改可以防止         │
│ 发布后改成广告           │
│                         │
│ 快照机制                 │
│ 本地记录和社区帖子       │
│ 是独立的，互不影响       │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 为什么不提供举报功能？   │
│                         │
│ 这个 app 不提供举报功能。│
│                         │
│ 不是因为不在乎内容质量， │
│ 而是因为现实。           │
│                         │
│ 第一个原因：             │
│ 一人开发，处理不过来。   │
│                         │
│ 举报功能需要人工审核，   │
│ 但这是一个人的项目，     │
│ 没有审核团队。           │
│                         │
│ 如果做了举报功能，       │
│ 却无法及时处理，         │
│ 反而会让用户失望。       │
│                         │
│ 第二个原因：             │
│ 举报功能容易被滥用。     │
│                         │
│ 恶意举报会导致：         │
│ 正常内容被误判隐藏       │
│ 用户之间的恶意攻击       │
│ 申诉处理不过来           │
│                         │
│ 没有足够的人力处理这些， │
│ 不如不做。               │
│                         │
│ 第三个原因：             │
│ 不做技术检测。           │
│                         │
│ 这个 app 不对内容进行    │
│ 自动检测和过滤。         │
│                         │
│ 原因：                   │
│ 技术检测容易误判         │
│ 影响用户体验             │
│ 依赖用户自律             │
│                         │
│ 发布到社区时会提醒用户   │
│ 不要包含隐私信息。       │
│                         │
│ 社区本身是只读的，       │
│ 即使有联系方式，         │
│ 也无法直接联系。         │
│                         │
│ 第四个原因：             │
│ 社区本身就是只读的。     │
│                         │
│ 即使有不当内容，         │
│ 也无法骚扰到你。         │
│                         │
│ 你可以选择不看。         │
│                         │
│ 如果遇到不当内容怎么办？ │
│                         │
│ 你可以：                 │
│ 不点进社区页面           │
│ 通过邮件联系开发者       │
│   （但不保证及时处理）   │
│                         │
│ 这不是完美的解决方案，   │
│ 但这是一个人能做到的     │
│ 最诚实的选择。           │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 为什么有时候设置了页面   │
│ 切换动画，进入时没有，   │
│ 反而退出时有？           │
│                         │
│ 这是 Flutter Web 的      │
│ 已知限制。               │
│                         │
│ 在 Web 平台上，页面进入  │
│ 动画经常被跳过或立即完成，│
│ 但退出动画是正常的。     │
│                         │
│ 这不是 bug，而是浏览器   │
│ 和 Flutter Web 渲染机制  │
│ 导致的技术限制。         │
│                         │
│ 如果你觉得这很影响体验， │
│ 可以在设置中选择         │
│ "无动画"模式。           │
│                         │
│ 移动端（Android/iOS）    │
│ 不存在这个问题，         │
│ 进入和退出动画都正常。   │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 为什么签到系统没有       │
│ 补签功能？               │
│                         │
│ 这个 app 有签到功能，    │
│ 但没有补签。             │
│                         │
│ 为什么？                 │
│                         │
│ 第一个原因：             │
│ 补签功能很难设计。       │
│                         │
│ 其他 app 的常见做法：    │
│                         │
│ 积分兑换补签券           │
│   → 阅读漫画 X 分钟      │
│   → 观看广告 X 次        │
│   → 完成每日任务         │
│                         │
│ 社交货币兑换补签券       │
│   → 点赞 XX 次           │
│   → 评论 XX 次           │
│   → 分享 XX 次           │
│                         │
│ 但 Serendipity 没有这些： │
│   漫画                   │
│   广告                   │
│   点赞                   │
│   评论                   │
│   分享                   │
│                         │
│ 这让补签功能的设计       │
│ 变得非常困难。           │
│                         │
│ 第二个原因：             │
│ 补签会破坏签到的意义。   │
│                         │
│ 签到的意义是什么？       │
│ 是每天的小仪式。         │
│                         │
│ 如果可以补签，           │
│ 用户会想：               │
│ "反正可以补，不急"       │
│                         │
│ 这反而降低了             │
│ 每日签到的动力。         │
│                         │
│ 第三个原因：             │
│ 符合产品调性。           │
│                         │
│ 这个 app 叫              │
│ "Serendipity"，          │
│ 核心是记录"错过"。       │
│                         │
│ 签到也可以"错过"，       │
│ 这本身就是一种           │
│ 接受遗憾的态度。         │
│                         │
│ 错过就错过吧，           │
│ 就像那些擦肩而过的       │
│ 缘分一样。               │
│                         │
│ 有些遗憾，               │
│ 注定无法弥补。           │
│                         │
│ 但至少，                 │
│ 我们记住了。             │
│                         │
│ 第四个原因：             │
│ 保持功能简洁。           │
│                         │
│ 不做补签功能：           │
│ 规则简单明了             │
│ 代码保持简洁             │
│ 维护成本低               │
│ 避免滥用                 │
│                         │
│ 连续签到天数更有价值，   │
│ 因为它是真实的。         │
│                         │
│ 如何避免忘记签到？       │
│                         │
│ 你可以：                 │
│ 开启签到提醒             │
│   （每天晚上 8 点）      │
│ 自定义提醒时间           │
│ 把签到当作睡前仪式       │
│                         │
│ 但如果真的忘了，         │
│ 那就接受吧。             │
│                         │
│ 这不是惩罚，             │
│ 而是提醒我们：           │
│ 有些事情，               │
│ 错过了就是错过了。       │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 为什么有时候登出时会     │
│ 看不到之前创建的         │
│ 记录/故事线？            │
│                         │
│ 这个 app 支持多账号，    │
│ 每个账号的数据是         │
│ 完全隔离的。             │
│                         │
│ 数据归属规则：           │
│                         │
│ 未登录时创建的数据       │
│ 归属于"首次登录的账号"。 │
│                         │
│ 场景1：未登录 → 注册账号A│
│                         │
│ 1. 未登录时创建3条记录   │
│ 2. 注册账号A             │
│ 3. 自动绑定：3条记录     │
│    归属账号A             │
│                         │
│ 场景2：未登录 → 登录账号A│
│        → 登出 → 登录账号B│
│                         │
│ 1. 未登录时创建3条记录   │
│ 2. 登录账号A             │
│ 3. 自动绑定：3条记录     │
│    归属账号A             │
│ 4. 登出（数据不清空）    │
│ 5. 登录账号B             │
│ 6. 账号B看不到那3条记录  │
│    （因为归属账号A）     │
│                         │
│ 场景3：账号A登出后       │
│        离线创建新数据     │
│                         │
│ 1. 账号A登出             │
│ 2. 未登录时创建2条新记录 │
│ 3. 登录账号A             │
│ 4. 自动绑定：2条新记录   │
│    归属账号A             │
│                         │
│ 场景4：账号A登出后       │
│        离线创建新数据     │
│        然后登录账号B      │
│                         │
│ 1. 账号A登出             │
│ 2. 未登录时创建2条新记录 │
│ 3. 登录账号B（不是账号A）│
│ 4. 自动绑定：2条新记录   │
│    归属账号B             │
│                         │
│ 核心原则：               │
│ "谁先登录，               │
│  离线数据就归谁"         │
│                         │
│ 为什么这样设计？         │
│                         │
│ 第一个原因：             │
│ 支持离线使用。           │
│                         │
│ 你可以在未登录时         │
│ 创建记录，               │
│ 登录后自动归属到         │
│ 你的账号。               │
│                         │
│ 第二个原因：             │
│ 数据隔离更安全。         │
│                         │
│ 每个账号只能看到         │
│ 自己的数据，             │
│ 不会混淆。               │
│                         │
│ 第三个原因：             │
│ 符合直觉。               │
│                         │
│ 你创建数据后登录，       │
│ 数据自然归属该账号。     │
│                         │
│ 如何避免数据丢失？       │
│                         │
│ 1. 创建数据后及时登录    │
│ 2. 不要频繁切换账号      │
│ 3. 记住哪些数据归属      │
│    哪个账号               │
│                         │
│ 如果找不到数据了？       │
│                         │
│ 数据一定在某个账号里，   │
│ 尝试登录所有账号查看。   │
│                         │
│ 注意：                   │
│ 数据一旦绑定到账号，     │
│ 就无法在登出状态下查看。 │
│                         │
│ 这不是 bug，             │
│ 而是多账号数据隔离的     │
│ 必然结果。               │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 为什么不支持手机号注册？ │
```
