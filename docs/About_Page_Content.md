## 关于 Serendipity

```
┌─────────────────────────┐
│ 关于 Serendipity        │
├─────────────────────────┤
│                         │
│ Serendipity 是一个记录  │
│ "错过"的 app。           │
│                         │
│ 不是为了重逢，           │
│ 而是为了记住那些         │
│ 转瞬即逝的瞬间。         │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 但这个 app 相信奇迹。    │
│                         │
│ 虽然成功率只有 0.01%，  │
│ 虽然 10,000 次错过中    │
│ 只有 1 次会成功，        │
│                         │
│ 但仍然相信，             │
│ 命运会给你第二次机会。   │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 五重防御机制             │
│                         │
│ 这个 app 做了匹配功能，  │
│ 但遇到骗子的可能性极低： │
│                         │
│ 第一重：对方也使用       │
│         Serendipity     │
│                         │
│ 第二重：对方也记录了     │
│         那次错过         │
│                         │
│ 第三重：双方都将记录     │
│         加入了故事线     │
│         （技术要求）     │
│                         │
│ 第四重：GPS 距离 < 200米 │
│         时间差 < 15分钟  │
│         地点名称模糊匹配 │
│                         │
│ 第五重：标签的双向验证   │
│         对方需要准确描述 │
│         你的特征         │
│         你也需要确认     │
│         对方的特征       │
│                         │
│ 骗子无法通过第五重验证。 │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 状态说明                 │
│                         │
│ 这个 app 有六种状态，    │
│ 核心区别在于：           │
│ 有没有交流过。           │
│                         │
│ 🌫️ 错过                 │
│ 第一次看到，但没说话     │
│                         │
│ 🌟 再遇                 │
│ 又看到了，但还是没说话   │
│ （命运的第二次机会）     │
│                         │
│ 💫 邂逅                 │
│ 终于说话了！             │
│ （第一次真正的见面）     │
│                         │
│ 💝 重逢                 │
│ 分开后再次相遇           │
│ （需先经历别离）         │
│                         │
│ 🥀 别离                 │
│ 主动结束了               │
│ （分手、好聚好散）       │
│                         │
│ 🍂 失联                 │
│ 被动消失，再也没见过     │
│                         │
│ 简单来说：               │
│ • 没说过话 → 错过/再遇  │
│ • 说过话了 → 邂逅        │
│ • 分开后又遇 → 重逢      │
│ • 主动结束 → 别离        │
│ • 被动消失 → 失联        │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 如何记录多次见面？       │
│                         │
│ 每次见面 = 一条新记录。  │
│                         │
│ 例如：                   │
│ • 2月8日在地铁看到 TA    │
│   → 创建记录1，状态"错过"│
│                         │
│ • 2月9日又看到 TA        │
│   → 创建记录2，状态"再遇"│
│                         │
│ • 2月10日终于说话了      │
│   → 创建记录3，状态"邂逅"│
│                         │
│ • 2月15日分手了          │
│   → 创建记录4，状态"别离"│
│                         │
│ • 3月1日又遇到了         │
│   → 创建记录5，状态"重逢"│
│                         │
│ 然后通过"故事线"功能，   │
│ 把这5条记录关联起来，    │
│ 形成完整的时间线。       │
│                         │
│ 不要在旧记录上修改状态！ │
│ "更改状态"功能只是用来   │
│ 修正手滑点错的情况。     │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 设计理念                 │
│                         │
│ 这个 app 不会告诉你      │
│ "正在匹配中"，           │
│ 因为那会让你焦虑。       │
│                         │
│ 如果根本没匹配上，       │
│ 不会收到任何通知，       │
│ 因为那会让你失望。       │
│                         │
│ 只会在匹配成功时，       │
│ 轻轻告诉你：             │
│ "也许，TA 就在这里。"    │
│                         │
│ 进入验证阶段后，         │
│ 如果验证失败，           │
│ 会温柔地告知你。         │
│                         │
│ 然后，选择权交给你。     │
│                         │
│ 你可以联系 TA，          │
│ 也可以继续错过。         │
│                         │
│ 因为有些美好，           │
│ 留在记忆里就够了。       │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 注意                     │
│                         │
│ 这个 app 的侧重点是      │
│ 帮助你记录错过的瞬间，   │
│ 而不是邂逅或重逢。       │
│                         │
│ 邂逅后如果继续在一起，   │
│ 就不需要再记录了，       │
│ 因为已经不是"错过"，     │
│ 而是"拥有"。             │
│                         │
│ 只有分开了（别离/失联），│
│ 才又回到"错过"的状态。   │
│ 这时如果再次相遇，       │
│ 就是"重逢"。             │
│                         │
│ 匹配功能更像是一个       │
│ 美好的愿望，             │
│ 因为现实是：             │
│                         │
│ 大多数人都输在了起跑线上。│
│                         │
│ 你用了这个 app 记录了TA，│
│ 但 TA 没有记录你。        │
│ 说明 TA 根本不在乎你。    │
│ 能匹配成功的前置条件是    │
│ TA也记录了你，所以概率只有0.01% │
│ 本质就是双向暗恋。        │
│                         │
│ 所以，                  │
│ 不要对匹配抱太大期望。   │
│                         │
│ 这个 app 的核心价值是：  │
│ 记录、回忆、接受遗憾。   │
│                         │
│ 有些错过，               │
│ 注定只能被记住。         │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 为什么树洞不能互动？     │
│                         │
│ 这个 app 有一个"树洞"   │
│ 功能，你可以匿名发布     │
│ 你的错过。               │
│                         │
│ 但树洞没有评论、点赞、   │
│ 也无法联系发布者。       │
│                         │
│ 为什么要这么"残酷"？     │
│                         │
│ 第一个原因：防骗。       │
│                         │
│ 如果树洞可以互动，       │
│ 骗子会怎么做？           │
│                         │
│ 1. 看到你的记录          │
│ 2. 假装就是那个人        │
│ 3. 主动联系你            │
│ 4. 骗取信任              │
│                         │
│ 零互动 = 骗子无法联系你。│
│                         │
│ 如果真的是 TA，          │
│ 匹配功能会帮你找到。     │
│ 如果不是，               │
│ 那就让它留在记忆里。     │
│                         │
│ 第二个原因：             │
│ 这才是"错过"的本质。     │
│                         │
│ 你发布到树洞后，         │
│ 也许 TA 会看到，         │
│ 也许 TA 会想'这不是我吗？'，       │
│ 但你永远不会知道。       │
│                         │
│ 这种"也许"的不确定性，   │
│ 才是最美的遗憾。         │
│                         │
│ 如果可以评论、点赞，   │
│ 那就不是"错过"了，       │
│ 而是"社交"。             │
│                         │
│ 第三个原因：             │
│ 这个 app 叫"错过了么"，  │
│ 不是"邂逅了么/重逢了么/  │
│ 失去了么"。              │
│                         │
│ 有些错过，               │
│ 注定只能是错过。         │
│                         │
│ 但至少，                 │
│ 我们记住了。             │
│                         │
│ 有人说这个设计很"刀"，   │
│ 但这就是错过的本质——     │
│ 有些遗憾，               │
│ 注定无法弥补。           │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 为什么不能修改发布到     │
│ 社区的记录？             │
│                         │
│ 发布到社区后，           │
│ 你可以删除，             │
│ 但不能修改。             │
│                         │
│ 为什么？                 │
│                         │
│ 第一个原因：             │
│ 保持真实性。             │
│                         │
│ 发布到社区的那一刻，     │
│ 就是你当时的真实感受。   │
│                         │
│ 如果可以随意修改，       │
│ 就失去了"错过"的         │
│ 真实性和即时性。         │
│                         │
│ 第二个原因：             │
│ 防止滥用。               │
│                         │
│ 如果可以修改，           │
│ 有人可能会：             │
│                         │
│ 1. 先发布正常内容        │
│ 2. 等别人看到后          │
│ 3. 修改成广告或联系方式  │
│                         │
│ 不允许修改 = 防止这种    │
│ 滥用行为。               │
│                         │
│ 第三个原因：             │
│ 社区发布是快照。         │
│                         │
│ 发布到社区时，           │
│ 会创建当前记录的快照。   │
│                         │
│ 之后你修改私人记录，     │
│ 不会同步到社区。         │
│                         │
│ 这样保持社区内容的       │
│ 真实性和一致性。         │
│                         │
│ 如果想更新社区内容？     │
│                         │
│ 你可以：                 │
│ • 删除旧的发布           │
│ • 重新发布新的内容       │
│                         │
│ 这样既保持了真实性，     │
│ 又给了你更新的机会。     │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 为什么不提供举报功能？   │
│                         │
│ 这个 app 不提供举报功能。│
│                         │
│ 不是因为不在乎内容质量， │
│ 而是因为现实。           │
│                         │
│ 第一个原因：             │
│ 一人开发，处理不过来。   │
│                         │
│ 举报功能需要人工审核，   │
│ 但这是一个人的项目，     │
│ 没有审核团队。           │
│                         │
│ 如果做了举报功能，       │
│ 却无法及时处理，         │
│ 反而会让用户失望。       │
│                         │
│ 第二个原因：             │
│ 举报功能容易被滥用。     │
│                         │
│ 恶意举报会导致：         │
│ • 正常内容被误判隐藏     │
│ • 用户之间的恶意攻击     │
│ • 申诉处理不过来         │
│                         │
│ 没有足够的人力处理这些， │
│ 不如不做。               │
│                         │
│ 第三个原因：             │
│ 不做技术检测。           │
│                         │
│ 这个 app 不对内容进行    │
│ 自动检测和过滤。         │
│                         │
│ 原因：                   │
│ • 技术检测容易误判       │
│ • 影响用户体验           │
│ • 依赖用户自律           │
│                         │
│ 发布到社区时会提醒用户   │
│ 不要包含隐私信息。       │
│                         │
│ 社区本身是只读的，       │
│ 即使有联系方式，         │
│ 也无法直接联系。         │
│                         │
│ 第四个原因：             │
│ 社区本身就是只读的。     │
│                         │
│ 即使有不当内容，         │
│ 也无法骚扰到你。         │
│                         │
│ 你可以选择不看。         │
│                         │
│ 如果遇到不当内容怎么办？ │
│                         │
│ 你可以：                 │
│ • 不点进社区页面         │
│ • 通过邮件联系开发者     │
│   （但不保证及时处理）   │
│                         │
│ 这不是完美的解决方案，   │
│ 但这是一个人能做到的     │
│ 最诚实的选择。           │
│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
│                         │
│ 技术选择                 │
│                         │
│ • Flutter + Dart        │

│                         │
│ ━━━━━━━━━━━━━━━━━━━━━  │
```

## 实现说明

### UI 布局建议
- 使用 `SingleChildScrollView` 包裹整个内容
- 每个章节之间用分隔线（━━━）区分
- 使用合适的字体大小和行距，确保可读性
- 考虑添加渐变背景或纹理增加氛围感

### 文案展示方式
- 可以使用 `Card` 或 `Container` 模拟手机界面的边框效果
- 文字居中对齐
- 使用温暖的配色（灰蓝色调）
- 适当的内边距（padding）

### 交互设计
- 页面顶部添加返回按钮
- 底部可以添加"联系开发者"按钮（打开邮件客户端）
- 考虑添加"分享"功能（分享 app 给朋友）
